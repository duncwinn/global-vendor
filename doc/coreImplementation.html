Global Vendor

Core Implementation

In this post I will discuss the api in a bit more detail, and then talk about the implementation of the core services.

Prerequisites

Before we get started, you will need to download the project and setup your environment. Make sure you have all of the following installed:

Required

  • A Java1.6 VM
  • Gradle
  • The Global Vendor Project itself

Optional

All the tests can be run from the command line, but you'll find it easier to use an IDE. The gradle scripts are set up with Eclipse/STS in mind, but should also work with Intellij.
  • Spring Tool Suite
  • Git Source Code Control

Advanced

You will need RabbitMQ for the more advanced configurations, but you don't need to install it just yet.
  • RabbitMQ
The test deployments use the embedded H2 database, but you can opt to use any other database you choose provided you supply the driver. The project comes with a configuration using MySQL.

In this post we will be discussing the gv-api and gv-core projects.

gv-api

This project defines the service interfaces used by the other projects. It also defines the message formats used by the SOAP Gateway and Spring Batch for file processing. The main packages in this project are as follows:
PackageDescription
gv.apiContains the model objects (DTOs) Warehouse, Product, Order etc
gv.distributor.apiThe service interface for operations which can be performed by distributors
gv.jaxbSupport classes to help with JAXB marshalling and unmarshalling
gv.orders.apiThe low-level interface to the order management system
gv.products.apiThe interface to the product management system
gv.stock.apiA trivial StockService interface
gv.testTwo marker interfaces (UnitTest and IntegrationTest) used with the @Category annotation to group JUnit tests
gv.warehouse.apiThe service interfaces which a warehouse must implement, including messaging DTOs
